# 전체 시스템 Task(Thread) 구조 설계 및 개발 로드맵

## 1. 전체 시스템 Task(Thread) 구조 설계

Zephyr OS는 멀티스레딩을 지원하는 RTOS(실시간 운영체제)입니다. 기능들이 서로 간섭하지 않고 독립적으로 동작하도록 업무를 여러 개의 Task(Zephyr에서는 **Thread**라고 부릅니다)로 나누고, 그 사이를 **메시지 큐(Message Queue, `k_msgq`)**·메모리 슬랩·링 버퍼 등으로 연결하는 구조가 유지보수와 실시간성 측면에서 안정적입니다. 특히 “수집 스레드는 빠르게 읽고 넘기기만”, “네트워크 스레드는 블로킹 송수신에 집중”처럼 역할을 분리하면, 한 경로의 지연이 다른 경로 전체를 붙잡는 현상을 줄일 수 있습니다.

현재 Smart Gateway 프로젝트의 요구사항(ADC 주기 수집·이더넷 UDP 전송·RS-232 Modbus·TCP 외곽 프레임 중계)에 맞춘 **권장 Task 구조**는 다음과 같습니다.

---

### Task 1: ADC 수집 스레드

**역할:** 설정된 주기(예: 수 ms)로 LPADC 채널을 순회하며 **다채널(운영 8채널, 시험 2채널 등)** 샘플 값을 읽어 옵니다. 윈도우 단위 min/max·스냅샷 갱신도 이 스레드에서 수행하는 것이 자연스럽습니다.

**동작:** 읽어 온 결과를 `adc_snapshot_t` 등으로 묶은 뒤, **UDP 전송 스레드가 소비할 수 있도록 `k_msgq`(UDP 전송용 메시지 큐)** 에 넣습니다. 큐가 가득 찰 경우의 정책(가장 오래된 항목 버리기, 카운트만 증가 등)은 제품 요구에 맞게 정합니다.  
*(참고: 기존 구현에서는 메시지 큐 대신 스핀락으로 보호하는 공유 스냅샷에 복사하는 방식을 쓰고 있으며, 추후 부하·지연 분석 후 큐 방식으로 옮길 수 있습니다.)*

---

### Task 2: RS-232 통신 스레드 (다채널)

**역할:** 한 개 또는 여러 개의 RS-232(UART) 포트에서 들어오는 바이트 스트림을 **인터럽트 기반**으로 수신합니다. Modbus RTU처럼 프레임 단위로 맞춰야 한다면, 이 스레드(또는 ISR에서 던지는 링 버퍼 소비측)에서 무음 시간·바이트 수로 프레임을 완성합니다.

**동작:** 수신·파싱된 **Modbus 페이로드 또는 원시 프레임**을 **TCP/IP 통신 스레드가 서버로 올릴 수 있도록 `k_msgq`(TCP/IP 전송용 메시지 큐)** 에 넣습니다. 반대로 TCP에서 “장비로 보낼 송신 데이터”가 오면, 이 스레드가 UART FIFO로 내보냅니다.  
*(참고: 게이트웨이가 “서버 요청 1건 → UART 송신 → 응답 수신 → TCP 응답”처럼 **동기 요청-응답**이면, RS-232를 전담 스레드로 두고 TCP 스레드는 큐/세마포로 “송신 허용·응답 도착”만 맞추는 식으로 설계할 수 있습니다. 현재 코드베이스는 TCP 스레드 컨텍스트에서 Modbus TX/RX를 직접 호출하는 형태도 혼재할 수 있으므로, 확장 시 Task 2를 분리하는 편이 명확합니다.)*

---

### Task 3: UDP 전송 스레드

**역할:** 이더넷 링크가 올라온 뒤, **서버(PC)로 ADC 스트림을 주기적으로 전송**합니다. MessagePack 등 애플리케이션 형식도 이 스레드에서 인코딩합니다.

**동작:** **‘UDP 전송용 메시지 큐’** 에서 ADC 스냅샷(또는 인코딩 전 구조체)을 꺼내, BSD Socket API(`zsock_socket` / `zsock_sendto`)로 **고정 피어 IP:포트**에 전송합니다. 수신이 필요하면 같은 소켓에 바인드하고, **TCP TIMESYNC(0x00) 이후에만 수신 루프를 돈다**는 식의 게이트 정책을 둘 수 있습니다.

---

### Task 4: TCP/IP 통신 스레드

**역할:** 상대(예: PC의 TCP 서버, SCADA, 시리얼 터미널 도구)와 **TCP 세션을 맺고 유지**합니다. 외곽 프로토콜(STX·Length·MsgType·Seq·Body·Error·ETX) 파싱·조립은 이 스레드에서 수행하는 것이 일관됩니다.

**동작:** 수신 스트림에서 완전한 프레임을 만든 뒤, 요청이면 RS-232 경로(Modbus)와 연동하고, 응답 프레임을 다시 TCP로 송신합니다. **RS-232 전담 스레드와는 메시지 큐 또는 명확한 동기화 객체**로 연결하여, 소켓 버퍼와 UART 버퍼가 서로 뒤섞이지 않게 합니다. 연결이 끊기면(클라이언트 모드) 재접속 루프로 복구합니다.

---

### 메시지 큐를 둘 때의 이득 (요약)

- ADC 수집이 네트워크 지연에 직접 묶이지 않음.  
- TCP 처리 시간이 길어져도 UART ISR은 링에만 적재하고, 상위는 큐 깊이로 밀림을 관측 가능.  
- 스레드 간 인터페이스가 “구조체 + 큐”로 고정되어 단위 테스트·로그 삽입이 쉬움.

---

## 2. 1단계 개발 로드맵: ADC → 이더넷 UDP 통신

가장 먼저 목표로 하는 **‘ADC 데이터의 이더넷 UDP 전송’** 을 구현하기 위해, 다음 순서로 진행하는 것을 권장합니다.

### 2.1 ADC 기능 구현

- **Device Tree:** 보드용 `*.overlay`(또는 보드 기본 DTS)에서 **LPADC 노드·채널·핀 mux** 가 애플리케이션과 일치하는지 확인합니다.  
- **Zephyr ADC API:** `adc_channel_setup`으로 채널을 설정한 뒤, `adc_read()`로 **1채널**부터 raw 값이 안정적으로 읽히는지 검증합니다.  
- **채널 확장:** `adc_inputs[]` 등과 맞추어 **8채널**(또는 제품 규격 채널 수)까지 확장하고, **동일 주기**로 순차 변환합니다.  
- **애플리케이션 데이터:** 전압 변환·2초(또는 규격) 윈도 min/max·타임스탬프가 포함된 **스냅샷 구조체**를 정의합니다. (추가로 Task 1과 Task 3 사이에 **메시지 큐**를 두면 1단계 설계와 자연스럽게 이어집니다.)

### 2.2 이더넷 네트워크 활성화

- **설정 파일(`prj.conf`):** `CONFIG_NETWORKING`, `CONFIG_NET_IPV4`, `CONFIG_NET_UDP`, `CONFIG_NET_SOCKETS`, `CONFIG_NET_L2_ETHERNET` 등 **네트워킹·소켓·이더넷** 옵션을 켭니다.  
- **주소 할당:** 개발 단계에서는 **고정 IPv4**(예: `CONFIG_NET_CONFIG_MY_IPV4_ADDR`)로 공유기 대역에 맞추거나, 운영 정책에 따라 **DHCP**를 켜고 부팅 로그에서 주소 할당 여부를 확인합니다.

### 2.3 UDP 소켓 프로그래밍 및 통합

- **BSD Socket API:** Zephyr의 `zsock_socket(AF_INET, SOCK_DGRAM, IPPROTO_UDP)`로 소켓을 만들고, `zsock_sendto`로 **PC(서버) IP·포트**에 페이로드를 보냅니다.  
- **페이로드:** 1단계에서는 raw 바이트 나열로도 되고, 제품 단계에서는 **MessagePack** 등으로 `adc_snapshot_t`를 직렬화합니다.  
- **통합:** Task 1(또는 동일 프로세스 내 루프)에서 수집한 데이터를 Task 3이 일정 간격으로 전송하도록 하고, 와이어샤크·PC 수신 프로그램으로 수신을 확인합니다.

---

## 3. 이후 단계(개략)

- **2단계:** RS-232 Modbus RTU 안정 수신(인터럽트·링·프레임 간격 `Kconfig`) 및 Task 2 분리·큐 연동.  
- **3단계:** TCP 외곽 프레임·핸드셰이크(0x80↔0x00)·TIMESYNC와 UDP 게이트 연동, 클라이언트/서버 모드 선택.  

세부 프로토콜과 설정 항목은 `docs/TCP_MODBUS_GATEWAY.md` 및 `Kconfig`를 참고하면 됩니다.
